# 34 多任务入口选择

如果你同时跑多个 quest，先别急着给每个任务都绑一个聊天入口。

聊天软件适合提醒、追问、做决策；Web 工作区更适合看全局状态、切换项目、检查文件和处理 5 个以上的任务。

## 最简单的建议

| 任务数量 | 推荐入口 | 说明 |
| --- | --- | --- |
| 1-3 个 quest | 任选 | Web、TUI、微信、QQ、Telegram、Feishu 都可以，怎么顺手怎么来。 |
| 微信 | 只能放一个主 quest | 微信适合个人提醒和快速回复，但当前不适合拿来管理很多 quest。 |
| QQ | 最多 5 个左右 | QQ 适合同时跟进几个任务，但再多会很容易混。 |
| 超过 5 个 quest | Web 为主 | 直接回到 Web 工作区看和操作，不要把聊天窗口当总控台。 |
| 超过 10 个 quest | Web 总览 + 关键通知 | connector 只用来收里程碑、异常和需要你决策的消息。 |

一句话：

```text
少量任务，聊天入口很舒服；任务一多，就回到 Web 里看总览。
```

## 为什么不建议全靠聊天管很多任务

聊天窗口的问题不是不能用，而是任务一多就容易乱：

- 你很难一眼看出哪个 quest 正在跑、哪个卡住了。
- 多个任务的消息混在一起，容易回错上下文。
- 自动绑定私聊时，最新活跃 quest 可能不是你以为的那个。
- 文件、日志、Canvas、实验证据还是在 Web 工作区里看最清楚。

所以，connector 更适合当“通知和快速协作入口”，不是 10+ quest 的主面板。

## 多任务时怎么设置

如果你同时跑多个 quest，建议这样做：

1. 用 Web 工作区作为主入口。
2. 重要 quest 才绑定 connector。
3. 不重要的 quest 不绑定聊天入口，只在 Web 里看。
4. 微信只绑定一个当前最重要的 quest。
5. QQ 最多绑定 5 个左右的常用 quest。
6. 任务很多时，减少 connector 主动推送，只保留关键里程碑和需要你决策的消息。

## 跟 `max_concurrent_quests` 的关系

`daemon.max_concurrent_quests` 只是控制最多同时活跃多少个项目，不代表你应该用聊天软件同时管理这么多项目。

如果你把这个值调大，建议同时做到：

- 确认机器资源足够。
- 确认 runner 配置稳定。
- 主要用 Web 查看状态。
- connector 只负责少量重要通知。

## 什么时候还需要外部 controller

大多数用户不需要外部 controller。

只有当你真的在长期跑很多 quest，并且希望自动做这些事时，才需要看 [19 External Controller 指南](./19_EXTERNAL_CONTROLLER_GUIDE.md)：

- 定期扫描所有 quest 状态。
- 自动找出卡住或跑偏的任务。
- 写一份汇总报告。
- 只把真正需要人看的消息发到 connector。

如果只是 1-5 个任务，用 Web 加一个常用聊天入口就够了。
